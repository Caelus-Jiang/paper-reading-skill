# 第一性原理论文速览

以第一性原理思考者的角色，从基本原理和常识出发推演论文的做事思路。把速览放在摘要之后、既有深度报告之前，但在完成原文阅读、主张—证据账本和 reviewer-level 分析之后再写。速览是深度分析的压缩索引，不是只读摘要后形成的预判。

## 推理纪律

- 从任务的对象、可观测信息、可控变量、目标、约束和最小必要假设出发，先去掉论文术语，再重建论文方案。
- 将 `Inspiration`、`Insight` 与 `Novelty` 分开：Inspiration 是触发洞见的既有事实、失败现象、理论或相关工作；Insight 是作者对问题结构或解法结构的判断；Novelty 是为落实 Insight 而新增的架构、方法或策略。
- 优先使用作者明确给出的动机链。作者未明确说明时，标记为 `reviewer 推断`，并给出原文依据；不得把事后合理化写成作者的真实思路。
- 每个重要判断绑定论文位置、公式、图表或实验。无法确认时直接写“原文未明确说明”或“证据链未闭环”。
- 省略客套话、背景铺陈和重复摘要，直接回答六问。

## 六问结构

### 1. Task

尽可能形式化地说明问题。至少回答：输入与已知信息是什么、输出或决策是什么、优化目标或成功标准是什么、有哪些约束与假设、训练与推理阶段分别可访问什么。若论文适合写成优化问题，使用纯文本表达，例如“在约束 C 下，选择策略 pi，使期望任务损失最小”。

### 2. Challenge

说明传统方法为何在该问题上失败，不只列举缺点。把困难还原到信息不足、优化困难、计算或存储成本、数据分布、信用分配、泛化、系统约束等基本原因，并指出论文用什么证据证明这些挑战确实存在。

### 3. Insight

逐条写清：

1. 哪个 Inspiration 触发了洞见；
2. Insight 的内容是什么；
3. 它是关于问题表示、信息结构、目标函数、优化、架构、训练、推理还是数据策略的洞见；
4. 多个 Inspiration 与多个 Insight 之间如何对应。

### 4. Novelty

判断创新属于架构、方法、目标、训练策略、推理策略、数据策略、系统实现还是已有组件重组。对每个 Novelty 严格使用：

```text
【创新点解决的问题是什么】 -> 【受哪个 Insight 启发】 -> 【设计了什么创新点，尽可能具体描述】
```

箭头两端写具体内容，不要原样保留模板占位语。若没有可独立成立的创新，不得硬凑；明确写“未识别到可独立成立的 Novelty”，并解释它更接近何种已有方法的组合、参数化或工程增强。

### 5. Potential flaw

依次回答：

1. 当前问题情境的维度、条件、约束、环境或任务覆盖有什么局限，架构延伸后可进入哪些新情境；
2. 在当前情境下，噪声、偏差、长尾、缺失、非平稳、相关性、标签错误、分布偏移或对抗性等坏数据性质会造成什么特有困难；
3. 哪个困难最值得发展成论文，并按影响面、严重性、可检验性和解法非平凡性说明理由。

### 6. Motivation

用一条或多条问句重建最自然的 idea 生成路径，典型形式为“之前的方法因为基本约束 X 而必须做 Y，那可不可以利用事实 Z，改为 W？”。从问题本质推到最小可行变化，不要从最终架构倒推一串不真实的灵感故事。

## 版式约束

- 本速览内部不使用任何形式的 LaTeX，也不使用美元符号数学定界符；公式和变量关系一律写成普通文本。
- 深度报告的后续章节仍按公式规范使用 LaTeX，本限制只作用于 `## 0. 第一性原理论文速览`。
- 保留 `Task`、`Challenge`、`Insight`、`Novelty`、`Potential flaw`、`Motivation` 标签，正文解释使用中文。
