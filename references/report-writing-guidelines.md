# 报告写作规范

## 目录

- 自包含原则
- 证据与语言
- 图片与表格
- 数学与代码
- Obsidian Markdown

## 自包含原则

唯一交付主体是 `output/{arxiv_id}_{title}/{arxiv_id}_阅读报告.md`。以下内容必须写进主报告：

- arXiv、hjfy、papers.cool 链接
- 核心主张及证据强度
- 主结果、关键消融和实验—主张矩阵
- 相关论文表与实际引用的外部文献
- 关键图片、公式及其就地解释
- 复现风险与代码—论文不一致

缓存 JSON 是证据索引，不能替代正文。

4.5 相关论文表只写论文标题、基础 arXiv ID、作者/年份、来源/类型、关系和概述，不添加网络链接。Obsidian 双链由同步器根据 vault 缓存索引自动注入；未命中时保持纯文本。外部核查链接仅放附录 B。

## 证据与语言

- 区分作者显式主张、隐含主张和 reviewer 推断。
- 每个重要判断绑定原文位置、公式、图表或外部文献。
- 证据不能闭环时直接写“证据链未闭环”。
- 不用“一定、证明了、完全解决”等超过证据强度的措辞。
- 内联枚举改成有序列表，每项用粗体关键词开头。

## 图片与表格

图片优先使用作者源码原图并与 PDF 核对，其次 PDF 嵌入图、figure 裁剪和整页 fallback。禁止网页截图。

图片按论证位置插入：任务图放 1.2，方法图放 2.3，结果图放 3.3，失败/消融图放 3.4 或对应质疑附近。每张图后解释展示内容、与主张的关系和证据边界。

正文分析依赖的关键表必须就地转写；附录只补完整列。不要只用“优于 baseline”代替数值。

## 数学与代码

- 行内数学使用 `$...$`，display 使用 `$$...$$`。
- 多行使用标准 LaTeX 环境。
- 需要引用的公式在块内用连续 `\tag{N}`。
- 数学不用反引号；函数、类、配置键和真代码使用反引号。
- 公式解释包含自然语言、符号、工程映射、假设和必要的 toy example。

## Obsidian Markdown

- 以 YAML frontmatter 开头，不含 `one_liner`。
- 不使用 H1；frontmatter 后立即写 `[!abstract]` callout。
- 默认图片语法为 `![alt](images/...)`。
- 外部 URL 用标准 Markdown 链接；只有 vault 内部笔记使用 wikilink。
- callout 用于摘要、关键结论、质疑和边界，每小节避免超过 2–3 个。
- 最终报告不得残留 TODO、隐藏调试注释或占位符。
